Current Truth

Facturel is a local Electron app for tracking bills and payment history without any cloud connectivity. The codebase was scaffolded in a single initial commit on 2025-12-04 and contains five React components (Dashboard.js, BillForm.js, BillDetails.js, PaymentForm.js, Sidebar.js), a SQLite database manager, and Electron main/preload files.

Every commit since that initial scaffold has been tooling-only: a major Electron version bump (v27 → v35 via Dependabot), a security fix for the tar-fs dependency, CI workflow updates for Claude code review and GitHub Actions, and a "checkpoint" commit on 2026-06-08 before restructuring for a common project framework. No feature work has happened in the src/ folder since December 2025.

The app's reference workflow was documented on 2026-03-27 using Chronicle Pro: open the app, review active bills, click through to each payee's website, pay the bill, then log the date/amount/notes. That six-step cycle is exactly what facturel needs to replace. Right now it is unknown whether the scaffold compiles and runs end-to-end, or whether the SQLite wiring persists data across restarts.

What It Does (Designed To)

Facturel lets privacy-conscious users manage recurring bills entirely on-device. Users add payees with names, due dates, payment URLs, logos, and notes; mark bills as monthly, quarterly, or yearly; and log each payment with date, amount, and notes. A tagging system groups bills (e.g. Utilities, Business); archiving keeps inactive bills out of the main view. Payment statistics show highest/lowest/average payments and yearly totals for comparison. A CSV export provides a manual backup path. All data lives in a SQLite file in the user's app-data directory — no login, no internet, no bank link required.

The target user is someone currently running a tool like Chronicle Pro who wants full local control without macOS-only lock-in. The app is cross-platform via Electron (Mac .dmg and Windows .exe).

The Reference Workflow (from 2026-03-27 diary)

This is what Rob does today in Chronicle Pro. Facturel needs to support this end-to-end before it becomes the actual tool.

  1. Open app → see list of active bills with last-payment context
  2. For each bill: review last payment date and amount
  3. Click through to payee website (payment URL)
  4. Pay the bill externally
  5. Return to app → log payment: date + amount + notes
  6. Repeat for all bills (Amex, CIBC, and others)

Diary note also flags "Details of credit card transactions" as a future enhancement — i.e., being able to see line items per payment, not just the top-level amount.

Known Gaps / Top 5 Backlog Items
  1. Payee URL quick-launch The core workflow step from the Chronicle diary — without it, facturel can't replace the existing tool. A clickable link on each bill row that opens the payee site in the system browser is the single highest-value addition.
  2. Recurring due-date calculation So the Dashboard shows bills sorted by "due soon" rather than creation order. Uses date-fns (already a dependency). Without this, the bill list has no urgency signal.
  3. Payment logging that persists across restarts The PaymentForm → SQLite → BillDetails chain needs to be verified and debugged end-to-end. If data doesn't survive a restart, the app is a toy.
  4. Bill list sorted by due date / overdue status Dashboard needs to surface what's overdue or coming up, not just a flat list. Directly tied to #2 — depends on due-date calculation being in place.
  5. CSV export So data isn't trapped if the user wants to switch tools. Also a trust signal — users who know their data is exportable are more likely to commit to the app.
